# WarTab Drag-and-Drop System — Full Audit

## 1. Executive Summary

**Overall health: 4/10** — Functional but unpredictable. The dominant failure mode is a **preview-visibility mismatch**: the insertion gap preview (cards shifting right) works for simple same-row inserts but breaks down for cross-row, multi-height scenarios. The highest-impact fix is a **computed-position gap preview** that shows the ACTUAL final position of the dragged card using `simulateGrid` to determine (row, col), then renders a card-shaped placeholder there — but clipped/positioned to NEVER overlap existing cards.

---

## 2. Critical Issues

### C1. No cross-row insertion preview
**What:** When dragging between rows, the preview simply doesn't render (the `else` branch hides `insertBar` and clears highlights). The user gets ZERO feedback.
**Severity:** `Critical`
**Why:** If you can't see where a card goes between rows, you can't reliably reorder. Users resort to trial-and-drop.
**Fix:** Add a between-row horizontal bar spanning the grid width at the gap center. Already partially there but missing when cursor is between rows.
**Code:**
```js
if(!targetRow && rows.length > 1){
  // Find which gap the cursor is in
  for(let i=0;i<rows.length-1;i++){
    const gapTop=rows[i].bottom, gapBottom=rows[i+1].top;
    if(e.clientY>=gapTop && e.clientY<=gapBottom){
      // Show horizontal bar at gap center
      insertBar.style.display='';
      insertBar.style.left=gr.left+'px';
      insertBar.style.top=((gapTop+gapBottom)/2-2)+'px';
      insertBar.style.width=gr.width+'px';
      insertBar.style.height='4px';
      insertBar.style.background='var(--accent)';
      insertBar.style.border='none';
      insertBar.style.borderRadius='2px';
      // Insert before first card of next row
      dragState._beforeCardId=rows[i+1].cards[0]?.dataset.cardId||null;
      computeDropShift(dragState._beforeCardId);
      return;
    }
  }
}
```
**Risk:** Low — adds feedback where currently none exists.

### C2. Push-preview only shifts same-row cards — ignores multi-row cascades
**What:** When inserting a card shifts a multi-row card right, that card's lower rows don't visually shift. The pushed card overlaps with cards below it.
**Severity:** `Critical`
**Why:** The `push-preview` transform only applies `translateX` to cards in the SAME `targetRow`. A 2×1 card pushed right overlaps cards in the next row.
**Fix:** After pushing same-row cards, also find cards in LOWER rows that collide with the pushed area and apply a combined `translate(x, y)` to visually clear space.
**Risk:** Medium — cascade math is complex; wrong transforms could look worse.

### C3. Ghost uses row height, not card height
**What:** The ghost's height is `targetRow.bottom - targetRow.top`, which is the full row height, not the card's height.
**Severity:** `High`
**Why:** A 1×1 card's ghost in a row with a 2×1 card is 280px tall. The user expects a 140px ghost.
**Fix:** Ghost height should be `ch * 140` (the card's grid footprint), capped by the row height.
**Risk:** Low — visual only.

### C4. No touch/mobile drag support
**What:** `pointerdown`/`pointermove`/`pointerup` should handle touch, but `startDrag` calls `e.preventDefault()` which may conflict with mobile scroll. No `touch-action: none` on cards.
**Severity:** `Critical`
**Why:** Mobile users can't reorder. The drag handle has `touchAction: 'none'` but the card body itself doesn't.
**Fix:** Add `touch-action: none` to `.card` during drag, restore on drop.
**Risk:** Low — matches existing pattern on drag handles.

## 3. Quick Wins

### Q1. Cursor states during drag
**What:** No `cursor: grabbing` set during active drag.
**Fix:** Add `.card.dragging { cursor: grabbing !important; }` to CSS.
**Risk:** None.

### Q2. `will-change: transform` on draggable cards
**What:** Cards animate via `transform` during push-preview and FLIP, but no `will-change`.
**Fix:** Add `.card { will-change: transform; }` for GPU-accelerated transforms.
**Risk:** None — limited to reasonable number of cards.

### Q3. Push-preview transition cleanup ties to `transition` property
**What:** Push-preview sets `transition: transform 0.12s ease` on cards, then FLIP animation doesn't override it — causing conflicting transitions.
**Fix:** Clear `transition` before FLIP captures positions, or use a specific `transition` that FLIP overrides.
**Risk:** Low — timing-only bug.

### Q4. Drop-shift highlight uses `!important`
**What:** `.card.drop-shift { border-left: 3px solid var(--accent) !important; }` — `!important` prevents other styles from competing but makes override harder.
**Fix:** Use higher specificity instead (`#card-grid .card.drop-shift`).
**Risk:** None.

## 4. Interaction Improvements

### I1. Insert-vs-swap distinction
**What:** Currently all insertions look the same — cards shift right and get `drop-shift`. No visual difference between "insert between" and "swap positions."
**Fix:** 
- **Insert between**: shift cards right (current gap preview)
- **Swap with**: when the dragged card overlaps a card's center, show a different highlight (e.g., background tint + border, no shift)
**Risk:** Medium — needs center-overlap detection.

### I2. Animated reorder on drop
**What:** FLIP animation works but uses `cubic-bezier(0.34,1.56,0.64,1)` (overshoot) — feels bouncy for a dashboard.
**Fix:** Use `cubic-bezier(0.16,1,0.3,1)` (standard ease-out) or `ease-out`.
**Risk:** Low.

### I3. No edge auto-scroll
**What:** When dragging near the viewport edge, the page doesn't scroll.
**Fix:** In `onDragMove`, if cursor is within 40px of viewport edge, auto-scroll the container.
**Risk:** Low.

## 5. Performance Summary

| Issue | Current | Should be |
|-------|---------|-----------|
| Bounding rect reads | Every `pointermove` (60fps) | Cache on `pointerdown`, re-read only on row change |
| DOM queries | `config.cards.find()` every move | Reference from `dragState` |
| `simulateGrid` per move | Yes — O(n²) on every tick | Throttle to every 100ms or pointer leave row |
| Layout thrash | Mixing reads (`getBoundingClientRect`) and writes (`style.transform`) in same handler | Batch writes via rAF |
| Push-preview cleanup | `querySelectorAll('.push-preview')` every move | Store refs in `dragState` |

### P1. Bounding rect caching
**What:** `buildRowMap` calls `getBoundingClientRect()` on every card on every `pointermove`.
**Fix:** Cache rects on drag start, invalidate when scroll/resize occurs.
**Code:**
```js
// In startDrag:
dragState._cachedRects = [...grid.children].map(el => ({el, rect: el.getBoundingClientRect()}));
// In buildRowMap, use cached rects:
function buildRowMap(grid, excludeEl){
  const items=dragState._cachedRects.filter(c=>c.el.classList.contains('card')&&c.el!==excludeEl);
  // use items[n].rect instead of calling getBoundingClientRect()
}
```
**Risk:** Medium — cached rects go stale on resize; need invalidation.

## 8. Confidence Assessment

**Library:** Native Pointer Events API (no library). Custom implementation with `simulateGrid` for position calculation.

**Framework:** Vanilla JS. No React/Vue/Svelte — DOM manipulation is imperative.

**Grid strategy:** CSS Grid with `grid-auto-rows: minmax(140px, auto)` and `grid-column: span N` / `grid-row: span N`. Cards are positioned via CSS Grid auto-placement, not explicitly.

**Key constraint:** Because CSS Grid auto-placement handles layout (not JS), the `simulateGrid` function is REQUIRED for predicting final positions. It's accurate for the grid algorithm but the VISUAL feedback (push-preview, placeholder) can't perfectly mirror it without re-flowing the DOM during drag.

**Recommendation:** Accept that the preview is limited to "gap indication + drop-shift highlights" for same-row inserts, and add explicit between-row indicators. Don't attempt to show the full final layout during drag — it requires DOM reflow which is too expensive for 60fps interaction. The FLIP animation on drop provides the smooth transition to the final state.
